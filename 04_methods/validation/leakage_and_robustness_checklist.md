# 泄漏控制与稳健性检查清单

## 为什么需要清单

生态遥感机器学习最容易出现“看起来很准，但其实泄漏”的问题。尤其是站点重复、时间邻近、空间邻近、归一化和超参数调优都会引入乐观偏差。

## 数据泄漏类型

| 类型 | 表现 | 规避方式 |
|---|---|---|
| 站点泄漏 | 同一站点样本进入训练和测试 | site-level split |
| 时间泄漏 | 使用未来信息构建当前特征 | 滞后窗口只用过去 |
| 空间泄漏 | 临近像元/站点跨训练测试 | spatial block CV |
| 归一化泄漏 | 用全数据 fit scaler | 只用训练集 fit |
| 调参泄漏 | 反复根据测试集调参 | 保留独立 test |

## 稳健性检查

1. Pooled metrics 和 site-level metrics 同时报。
2. 按 PFT、气候区、站点数量分层。
3. 对极端站点和异常年份做敏感性分析。
4. 比较 NEE、GPP、RECO 三个分量。
5. 做 same-backbone ablation。
6. 检查结果是否依赖少数站点。
7. 检查 HLS 有效像元比例和 footprint 覆盖。

## 我的项目最低标准

如果要把结果写进论文、报告或项目复盘，至少需要说明：

- 使用 site-level CV。
- normalization 只用训练站点拟合。
- FAT 与 fixed-grid 使用相同 backbone。
- 报告站点级和分量级结果。
- 解释 GPP/NEE/RECO 的差异和局限。
